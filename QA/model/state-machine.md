# State architecture (Zustand stores + React contexts)

> Generated by QA audit 2026-07-21.

**Summary:** 53 Zustand stores under /store plus 3 React contexts. 44 stores use persist() via a shared SSR-safe/guarded AsyncStorage wrapper (getPersistStorage); 8 are ephemeral. Only 13 stores use partialize; the rest persist their whole state object. 40 of 44 persisted stores ship a no-op passthrough migrate; only 7 have real logic (settingsStore at v19 is by far the largest). A curated 34-key allowlist (services/cloudSync/snapshot.ts) is backed up to cloud/local/server. The highest data-loss risks are concentrated in: roundStore's compaction (silently strips shots/photos from rounds older than 50) and the asymmetric backup merge (round history unions on restore, but grow-mostly learned stores like caddie-memory/club-stats are blind-overwritten on restore while only protected on upload), plus the 0.6-length grow-mostly heuristic that can resurrect deleted data.

## State Architecture Map — SmartPlay Caddie

Read-only analysis of `/Users/timothyg/smartplay/store` (53 stores) and `/Users/timothyg/smartplay/contexts` (3 contexts). Focus: persistence shape and data-loss/corruption risk (this subsystem has a documented backup data-loss history).

### 1. Persistence plumbing (shared spine)

- **Backend:** `services/ssrSafeStorage.ts` → `getPersistStorage()`. Returns a `noopStorage` during SSR (`typeof window === 'undefined'`) and a `guardedStorage` (real AsyncStorage) on device. The guard wraps `setItem` so a quota/disk/OS-denied write rejection is logged to the issue log (`persist_write_failed`) instead of silently swallowed, then re-thrown so Zustand still sees the failure. It skips its own `issue-log-v1` key to avoid a write→fail→log→write recursion.
- **All persisted stores** call `createJSONStorage(() => getPersistStorage())` — **except `voiceHitRateStore`, which wires `AsyncStorage` directly** (bypasses both the SSR noop and the failure-logging guard). Minor inconsistency, low blast radius.
- **Backup path:** `services/cloudSync/` — `snapshot.ts` (gather/apply/union), `cloudBackup.ts`, `localBackup.ts`, `serverBackup.ts`, `autoBackup.ts`, and server side `api/backup.ts`. Backup operates on the **raw AsyncStorage blobs** of an explicit allowlist, not on the stores directly.

### 2. Persisted vs ephemeral

**Ephemeral (no persist(), in-memory only) — 8:** `connectivityStore`, `ghostStore`, `listeningSessionStore`, `screenshotModeStore`, `restModeStore`, `toastStore`, `smartVisionSignalStore`, `toolsMenuStore`. These are transient UI/session state and safe to lose.

**Persisted — 44** (detailed below).

### 4. Migration inventory

Real (non-passthrough) migrates — the only ones that can transform or drop data:

- **settingsStore (v19)** — the big one. Cumulative `version < N` seeding blocks (caddieAssignments seeding from old single caddiePersonality, accessibility defaults, persona-intensity floor repair, Harry→Kevin remap, voiceOnPhoneSpeaker default flip, etc.). Additive/repair only; no wipes observed. Note the **name mismatch**: key is `settings-store-v2` but schema `version: 19` — cosmetic, versions are tracked independently of the key suffix.
- **roundStore (v1)** — only fires for `version === 0`: back-fills `outcome`/`penalty_strokes` on shots and history. Additive.
- **caddieMemory (v2)** — normalizes to `{ players, courseBook }` with `?? {}` fallbacks. Safe.
- **trustLevel (v2)** — legacy L4/L5 users collapse to L2. Intentional value remap.
- **smartFinder (v2)** — `mode:'standard'` → `'target'`. Safe.
- **acousticCalibration (v2)** — seeds `targetSamples: []`. Additive.
- **issueLog (v2)** — back-fills `kind:'user'` on entries. Additive.

Everything else is `(s) => s as never` — an intentional "protect on version bump, never silent-wipe" pattern noted in the 2026-06-15 audit comments.

### 5. Backup allowlist (services/cloudSync/snapshot.ts)

`BACKED_UP_STORE_KEYS` = 34 explicit keys (allowlist, not "everything-minus"). Excluded by decision: `cage-store-v1`, `custom-caddie-media-v1` (media, phase-2 sync), and diagnostics/in-flight (`issue-log`, `conversation-log`, `gps-health`, `undo-mark`, `capture-engine`). `practice-plan-v1` and the voice-telemetry stores are also not backed up.

**Merge/union model:**
- `gatherSnapshot()` = multiGet of the allowlist (empty strings dropped).
- `mergeRoundBlobs()` unions `roundHistory` by `id` (confirmed: RoundRecord has `id: string` + `endedAt: number`), newer `endedAt` wins, shot-fuller copy wins ties, local appended last so its fuller copy survives a tie.
- `unionSnapshots()` (UPLOAD): round history unioned; `GROW_MOSTLY_KEYS` (23 learned stores) keep the cloud copy if the incoming blob is < 60% of cloud length.
- `applySnapshot()` (RESTORE): only round-store is unioned with local; all other keys are `multiSet` (blind overwrite).

### 7. Flagged data-loss / corruption risks (ranked)

1. **roundStore compaction silently strips per-shot + media from rounds older than 50** (`compactHistoryForPersist`, FULL_DETAIL_ROUNDS=50, roundStore.ts:667). On every persist write, rounds beyond the newest 50 have `shots`, `round_photos`, `emotionalLog`, `health` set empty/undefined. This is permanent once written and then propagates to the backup. Deliberate (bounds the single-row blob under Android's ~2MB read limit) but it IS irreversible shot-level loss for deep history.

2. **Asymmetric restore protection — grow-mostly stores are blind-overwritten on restore.** `unionSnapshots` protects the 23 `GROW_MOSTLY_KEYS` on UPLOAD, but `applySnapshot` only unions `round-store-v1`. Restoring an OLDER cloud backup onto a device with NEWER learned data (caddie-memory, club-stats, player-profile, practice-points, etc.) overwrites the richer local copy with the older cloud copy. Only rounds survive; all other learned CNS data can regress on restore.

3. **The 0.6-length grow-mostly heuristic can resurrect deleted data / reject legit shrink.** If a user legitimately shrinks a grow-mostly store (deletes clubs from the bag, prunes guests/family, resets practice), the upload union rejects the smaller-but-correct local blob and keeps the larger stale cloud copy (`nl < pl * 0.6`). Length is a crude proxy for richness.

4. **deleteRound does not propagate through the round union.** `mergeRoundBlobs` unions by id and re-adds any round present in either side. A round deleted locally but still in the cloud copy is resurrected on the next restore (and vice versa on upload). Deletions are not tombstoned.

5. **Whole-state persistence (31 no-partialize stores) rehydrates transient flags.** Stores without partialize persist in-flight/loading/session fields verbatim (functions are dropped by JSON, but data flags are not). A store that mutates a transient flag mid-operation and crashes will rehydrate that stale flag. roundStore's own history is a long trail of exactly this class of bug — three audit passes added missing in-round fields to its partialize because a mid-round crash dropped them.

6. **voiceHitRateStore bypasses the guarded storage** (direct `AsyncStorage`), so a persist-write failure there is neither logged nor SSR-safe. Low value store, low impact, but breaks the invariant.

7. **cageStore + customCaddieMedia are never backed up** — swing clips and custom caddie portraits are lost on reinstall by design (media reference local file:// paths). Expected, but worth stating: these are the two large persisted stores (cageStore ~79KB source) with zero cloud durability.

## Known risks
- roundStore.compactHistoryForPersist permanently strips shots/photos/emotionalLog/health from rounds older than the newest 50 on every persist write, then that loss propagates to backup
- applySnapshot (restore) only unions round-store-v1; the other 23 GROW_MOSTLY learned stores (caddie-memory, club-stats, player-profile, practice-points, etc.) are blind-overwritten, so restoring an older cloud backup regresses newer local learned data
- unionSnapshots grow-mostly guard uses a 60%-length heuristic: a legitimate shrink (user deletes clubs/guests/resets) is rejected and the stale larger cloud copy is kept, and deleted data can be resurrected
- mergeRoundBlobs unions rounds by id with no tombstone, so a round deleted on one device is re-added from the other side on the next restore/upload
- 31 persisted stores have no partialize and persist whole-state including transient/in-flight flags; a mid-operation crash rehydrates stale flags (the documented roundStore mid-round field-loss bug class)
- voiceHitRateStore wires AsyncStorage directly instead of getPersistStorage, bypassing SSR-safety and the persist-write-failure logging guard
- cageStore and customCaddieMedia (the two large media stores) are excluded from all backup, so swing clips and custom portraits are unrecoverable after reinstall
- settingsStore is at schema version 19 with a large cumulative migrate but persists under the key name settings-store-v2 (name/version mismatch is cosmetic but confusing during audits)
