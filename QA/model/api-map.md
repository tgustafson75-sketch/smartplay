# API surface (server /api + client /app/api)

> Generated by QA audit 2026-07-21.

**Summary:** The backend is 54 Vercel serverless routes under /Users/timothyg/smartplay/api plus 12 Expo Router dev-twin routes under /Users/timothyg/smartplay/app/api (the map's "11" undercounts by one — there are 12). Nearly every route is an AI-inference endpoint calling paid providers (OpenAI, Anthropic, Google Gemini, Deepgram) via the shared api/_aiProvider.ts abstraction, plus proxies to golfcourseapi, Google Maps, OpenWeatherMap, Overpass, OpenTopoData, and Golfbert. CORS was hardened in api/_cors.ts (audit S1) from wildcard to an origin allowlist, but only 6 of 54 routes actually import applyCors; the other 48 set no CORS headers at all (safe from browser response-reads, but preflight-exempt requests can still fire server-side work). The dominant risk: essentially all paid AI endpoints are unauthenticated — only pipecat-turn/pipecat-tool require a shared secret (and even that is bypassed if the env var is unset), backup uses an email+passphrase-derived key with IP throttling, and messages/usage validate email shape but have no real auth.

## API Surface Map — SmartPlay Caddie

Two route trees:

- **`/Users/timothyg/smartplay/api/*.ts`** — 54 files. These are the **canonical Vercel serverless functions** the production mobile app hits directly (via `getApiBaseUrl()` → `api.smartplaycaddie.com`). 4 are shared helpers (`_aiProvider`, `_cors`, `_kevinVoice`, `_supabase`) + `tsconfig.json`, leaving ~49 real routes.
- **`/Users/timothyg/smartplay/app/api/*+api.ts`** — 12 files (map said 11; actual count is 12). These are **Expo Router dev-server twins** that run ONLY in dev when `EXPO_PUBLIC_API_URL` is unset (clients fall through to `localhost:8081/api/...`). In prod they are dead code; `kevin+api.ts` is explicitly marked DEPRECATED TWIN and lags the canonical file.

### CORS model (`api/_cors.ts`)
`applyCors(req,res)` echoes `Access-Control-Allow-Origin` **only** for an allowlist (`smartplaylight.vercel.app`, `smartplay-light.vercel.app`, `smartplaycaddie-light.vercel.app`, `smartplaycaddie.com`, `www.smartplaycaddie.com`, `localhost:8081`, `localhost:3000`). Preflight from an allowed origin → 204; disallowed → 403. Native app sends no Origin, so it is unaffected. **This is the S1 fix that replaced `Access-Control-Allow-Origin: *`.**

**Only 6 routes actually call `applyCors`:** `course-places`, `hole-scan`, `kevin`, `swing-analysis`, `transcribe`, `course-ai-search` (partial). **The other ~48 routes set no CORS headers at all** — they rely on being native/same-origin only. No wildcard `*` remains anywhere. Client twins in `app/api` set no CORS either (dev-only).

### Auth model
There is **no bearer/session auth on the AI inference endpoints**. Gating that exists:
- `pipecat-turn` / `pipecat-tool` — shared secret `PIPECAT_SESSION_SECRET` via `x-pipecat-secret` header / `body.secret`; mismatch → 403. **Caveat: guarded as `if (SESSION_SECRET && incoming !== SESSION_SECRET)` — if the env var is unset, auth is silently skipped (fail-open).**
- `backup` — identity = `sha256(lowercased-email :: passphrase)`; passphrase never stored; wrong passphrase indistinguishable from no-backup; per-IP read throttle (audit S3) against brute force.
- `messages` / `usage` — validate email shape (regex blocks PostgREST filter injection) but **no authentication**; degrade to `200 ok:false` when Supabase unconfigured.
- `owner-triage` — comment explicitly says **"this route is unauthenticated"**; only the client UI hides the trigger behind `isOwnerEmail()`.

## Known risks
- Essentially ALL paid AI-inference endpoints are unauthenticated (kevin, swing-analysis, voice-intent, all vision analyzers, course-content, owner-triage, etc.). Anyone who knows the URL can drive unlimited paid Anthropic/OpenAI/Gemini/Deepgram spend on Tim's account. This is the standing 'S1 unauth wildcard-CORS paid endpoints' concern — the CORS half was fixed but the auth half was not.
- CORS allowlist (applyCors) is only wired into 6 of ~49 routes. The other ~43 (including expensive vision routes like swing-analysis's siblings ball-path, club-path, junior-swing-analysis, cage-review, lie-analysis) set no CORS headers — browser response-reads are blocked, but a cross-site POST with a CORS-simple content type can still trigger the server-side paid work (fire-and-forget) since CORS does not stop the request from executing.
- pipecat-turn / pipecat-tool auth is fail-open: `if (SESSION_SECRET && incoming !== SESSION_SECRET)` means if PIPECAT_SESSION_SECRET is unset in the environment, the 403 check is skipped entirely and the voice-brain endpoints become fully open.
- owner-triage.ts is documented as unauthenticated — the only gate is the client UI hiding the button behind isOwnerEmail(). Anyone can POST arbitrary context and consume Anthropic Sonnet.
- health.ts (GET, unauth) fires live probe calls to Anthropic, OpenAI, and Google on every request — a cheap-but-nonzero paid-call amplification / provider-status disclosure vector open to anyone.
- backup.ts protects data only by sha256(email::passphrase) with a per-IP throttle; a known email plus a weak passphrase and distributed IPs remains brute-forceable (acknowledged in-file, audit S3).
- messages.ts and usage.ts write to Supabase with only email-shape validation and no authentication — anyone can forge 'from' and inject messages or spam usage_events under any email identity.
- The X-AI-Provider request header lets clients force a provider (providerFromHeaderSafe in _aiProvider.ts); prior audit A1 noted an unsupported provider value could 500 ~10 routes. Header-driven provider selection on unauth endpoints is an availability/cost lever.
- app/api/kevin+api.ts is a DEPRECATED twin with known drift (no tier classifier, no vision, no OpenAI fallback); if EXPO_PUBLIC_API_URL is ever empty in a shipped/OTA build, clients silently fall back to this weaker handler — matches the recurring 'EXPO_PUBLIC empty in OTA' failure mode in memory.
- The task said 54 server + 11 client routes; actual counts are ~49 real server routes (54 files incl. 4 shared helpers + tsconfig) and 12 client twins, so any downstream tally based on 54/11 is off.
