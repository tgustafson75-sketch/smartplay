# Voice (Pipecat pipeline: capture → STT → brain → TTS)

> Generated by QA audit 2026-07-21.

**Summary:** The voice caddie is a tap-or-VAD audio capture pipeline that transcribes speech (Deepgram STT), classifies/routes it locally-first, and answers via one of two brains — the DEFAULT "pipecat" path (api/pipecat-turn, a provider cascade OpenAII→OpenAI→Gemini) or the legacy "kevin" path — then speaks the reply through a single-voice TTS layer (OpenAI gpt-4o-mini-tts with device-Speech fallback). It runs entirely on Vercel serverless behind one branded host (api.smartplaycaddie.com); no Railway/Python server is live (that is Phase-3 scaffold). Timeouts are deliberately layered so the client budget always exceeds the server cascade. All primary files are under the VOICE PATH FREEZE.

### 2. States (idle / listening / thinking / speaking)

The UI voice state is a 4-value enum from `components/CaddieAvatar` (`VoiceState`), driven through `wrappedOnVoiceStateChange` (which also flips `isThinking`). Both hooks emit the same four states.

- **idle** — nothing in flight; terminal state after a reply, error, or sign-off.
- **listening** — mic capturing (tap-hold, VAD active, or a re-opened follow-up mic).
- **thinking** — audio captured; STT + brain round-trip in flight.
- **speaking** — TTS playing (cloud mp3 or device fallback).

**Typical happy-path transition (pipecat):**
```
idle → (mic tap / VAD endpoint) → listening
     → processAudioUri: STT → thinking
     → processTurn: POST /api/pipecat-turn → thinking
     → speak(response) → speaking
     → idle  (then optional auto-listen re-arm → listening)
```

Re-entrancy guards:
- `usePipecatVoice.turnInFlightRef` — one brain turn at a time; a re-entrant `processTurn` is dropped (prevents double-award points / racing the shared history).
- `useVoiceCaddie.isProcessingRef` — released BEFORE the pipecat `await` so a follow-up re-listen isn't blocked (the "asks but doesn't listen" fix).
- VAD `startingRef` / `stopTokenRef` — guard concurrent recorder starts and cancel a start-in-flight when stop/disable lands.

### 3. Data flow (end-to-end)

1. **Capture** — `useVoiceCaddie.handleMicPress` (tap) OR `useVoiceActivityDetection.finishRecording` (auto-listen) produces an m4a URI. `stopSpeaking()` is called first so the caddie stops talking before it listens (barge-in).
2. **STT** — `processAudioUri` POSTs multipart audio to `/api/transcribe` → Deepgram Nova-2 (`nova-2`, smart_format, utterances, golf keyword boost, regex substitutions like "spark motion"→SmartMotion). Empty transcript → back to idle.
3. **Local-first precheck** — `services/localStatusResponder.tryLocalReply` answers deterministic status queries (yardage, hole, par, score, club, handicap) from round state + GPS with **zero network**. AI-led judgment types (what-should-I-hit, plays-like, can-I-reach) intentionally fall through to the brain.
4. **Intent router** — `voiceCommandRouter.route` calls `/api/voice-intent` (structured classifier, ~35 intents). A confident non-conversational hit executes a handler and returns. Disruptive tool-opens (record_swing, open_*, navigate, configure_drill) require `confidence: 'high'` to fire from the fast classifier (prevents narrative like "irons today" from yanking a screen open).
5. **Brain (conversational fallthrough)** — only conversational/unknown reaches here:
   - **pipecat mode** → `processTranscriptOverride` = `usePipecatVoice.processTurn` → `/api/pipecat-turn`.
   - **kevin mode** → `sendToBrain` (legacy `/api/kevin`).
6. **TTS** — `speak()` (cloud mp3) with device-Speech fallback; `onKevinSpoke` surfaces text to UI.
7. **Auto-listen re-arm** — if the reply ends as a question (`endsAsQuestion`) or continuous mode is on, and the user did NOT sign off (`isCloseIntent`), the mic re-opens after ~500ms.

**`/api/pipecat-turn` internals:** builds a rich system prompt (player, active-round yardage/score/shots, mental-state/spiral coaching, learned CNS memory, KB RAG, app-feature catalog, confirm-before-open + parse-everything rules). Runs `runAgenticLoop` with the full Kevin tool set. UI tools return as `tool_actions` for the RN client to dispatch; `lookup_course`/`lookup_hole`/`navigate` execute server-side.

### 4. Timeouts (layered so client budget > server cascade)

| Constant | Value | Location | Purpose |
|---|---|---|---|
| `TRANSCRIBE_TIMEOUT_MS` | 12 s | useVoiceCaddie | warm STT client abort |
| `COLD_TRANSCRIBE_TIMEOUT_MS` | 22 s | useVoiceCaddie | first-turn cold-boot STT (gated on `!isConnectionWarmed()`) |
| `DG_TIMEOUT_MS` | 12 s | api/transcribe | Deepgram server-side abort |
| `BRAIN_FETCH_TIMEOUT_MS` | 30 s | constants/voiceTimeouts | client→brain outer bound (kevin path) |
| `TURN_TIMEOUT_MS` | 35 s | usePipecatVoice | pipecat client abort (> server cascade) |
| `CAP_MS` / `FAST_FAIL_MS` | 9 s / 5 s | api/pipecat-turn | per-provider cap; slow-fail skips the retry |
| Provider cascade worst case | ~27 s (3×9s) | api/pipecat-turn | OpenAI→OpenAI→Gemini |
| processAudioUri STT (pipecat) | 20 s | usePipecatVoice | its own transcribe abort |
| `MIC_SILENCE_TIMEOUT_MS` | 1200 ms | useVoiceCaddie | endpoint quickly on silence |
| VAD `SILENCE_DURATION_MS` | 2800 ms | useVoiceActivityDetection | tolerate conversational pauses |
| OpenAI TTS SDK | 25 s timeout, 1 retry | api/voice | — |
| TTS text cap | 4000 chars | api/voice | — |

**Rule enforced everywhere:** the client timeout must strictly exceed the full server cascade so a HEALTHY-but-slow turn is never cancelled (pipecat 35s > 27s cascade; brain 30s > 27s; cold STT 22s > 12s warm).

### 5. Failover / hosts

- **Single branded host:** `PRIMARY_HOST = https://api.smartplaycaddie.com` (`services/apiBase.ts`). The `*.vercel.app` fallback was **removed** — networks content-filter `*.vercel.app` by name (the recurring "voice breaks again" root cause), so cross-host self-healing failover to that alias was actively harmful and is disabled. Any future 2nd host must be another clean custom domain.
- All fetches resolve through `getApiBaseUrl()` at fetch time (EXPO_PUBLIC_API_URL is empty in OTA bundles — the api-base-url spine bug; hardcoded prod fallback prevents relative "Invalid URL").
- `isConnectionWarmed()` / `markConnectionWarmed()` gate cold-vs-warm timeouts; `ensureBackendReachable` retries a warm ping with backoff.
- **Brain-level failover (in `/api/pipecat-turn`):** provider plan `['openai','openai','gemini']`; a fast OpenAI blip earns one retry, a slow (hung-the-cap) failure skips straight to Gemini. All providers missing → **graceful 200** ("Give me one sec and ask me again."), never a 502 (a 502 would trip the client circuit breaker as if the network died).
- **Client-side degrade (`usePipecatVoice`):** on non-ok / thrown fetch → try `answerOffline` (round state + golf KB, device-TTS) → else `captureOfflineStatement` + device-voice confirm ("No signal right now, but I saved that…"). Never double-processes (no legacy fallthrough after pipecat spoke).

### 6. One-voice invariant

Cloud/mp3 playback and device-TTS (`expo-speech`) are separate subsystems that must never overlap (`services/voiceService.ts`):
- A monotonic `speakGeneration` / `currentSpeechId` is bumped on every `speak()` and `stopSpeaking()`; queued-but-not-started bodies check the generation and skip.
- `stopSpeaking()` stops+unloads `currentSound` AND calls `Speech.stop()` — covers both subsystems, near-noop when idle.
- The device-TTS fallback path itself first stops any in-flight cloud sound before speaking, and re-checks `currentSpeechId` after the async stop ("two voices racing" fix).
- `isSpeaking()` tracks only the cloud sound, so barge-in code always calls `stopSpeaking()` unconditionally (it covers device-TTS too).
- Every capture start (`stopCapture`/mic open) calls `stopSpeaking()` first so opening the mic silences the caddie.
- Prewarm is a single idempotent, 30-s-deduped `prewarmVoice()` heartbeat (`services/voiceWarmup.ts`), fired on foreground and on mic tap; it warms TTS SDK + brain Lambdas (`{mode:'warmup'}` on `/api/voice`, `/api/voice-intent`, `/api/pipecat-turn`).

---

### 7. Warmup surfaces
`{ mode: 'warmup' }` is handled by `api/voice.ts` (runs a real 1-space TTS to hot the OpenAI SDK+TLS), `api/voice-intent.ts` (classifier ping), `api/pipecat-turn.ts` (`completeText` ping), and `api/transcribe.ts` (no-op — Deepgram is external, never cold).

---

### 8. Auth model
- `api/pipecat-turn.ts` & `api/pipecat-tool.ts` gate on `PIPECAT_SESSION_SECRET`, but only enforce when BOTH sides have a secret (OTA clients send `''`, so field calls aren't blocked — the secret is effectively unenforced in prod).
- Prompts carry a SECURITY POLICY preamble treating all client context (names, notes, history) as DATA, not instructions.

---

### 9. Phase-2 vs Phase-3
- **Phase 2 (LIVE):** text-in/text+tools-out over HTTPS POST to `/api/pipecat-turn`. No Railway/Python. This is the default production path.
- **Phase 3 (SCAFFOLD only):** `openSession`/`closeSession`/WebSocket/`pushGpsUpdate`/`pushHoleTransition` in `usePipecatVoice` + `api/pipecat-tool.ts` target a future Python Pipecat server (`pipecatServerUrl`). Inactive unless that URL is set; WS ping keep-alive is 25 s (Railway drops idle at 30 s).

## Known risks
- Pipecat secret auth is effectively unenforced in prod: /api/pipecat-turn only rejects when BOTH sides have a secret, and OTA clients send an empty string, so PIPECAT_SESSION_SECRET provides no real gate on the paid brain endpoint (matches audit finding S1 on unauth endpoints).
- voice-intent.ts and app/api/voice-intent+api.ts are hand-maintained LOCKSTEP TWINS (~655 lines each); any drift silently breaks voice classification in production. No test enforces parity — only a comment reminding to git diff both.
- Two parallel brains exist (pipecat vs legacy kevin) plus two prompt builders (api/pipecat-turn buildSystem vs api/kevin). Comments note repeated hand-syncing to keep 'parity'; this is a standing drift risk (spiral/emotional blocks, tool defs, fallback order must be kept identical by hand).
- The 35s (pipecat) / 30s (kevin) client timeouts are tuned to sit just above a ~27s 3-provider server cascade; if provider count, CAP_MS, or the cascade order changes, a healthy-but-slow turn can be cancelled again (the documented recurring 'first-try failure' regression).
- STT and brain both depend on a SINGLE host (api.smartplaycaddie.com) with cross-host failover deliberately disabled; if that one custom domain is unreachable, the only recourse is offline caddie / device-TTS capture — there is no second clean-domain fallback wired.
- usePipecatVoice.processAudioUri uses its own 20s transcribe timeout while useVoiceCaddie uses 12s/22s cold-aware — the pipecat audio path does NOT get the cold-boot 22s treatment gated on isConnectionWarmed(), so a first cold turn through the pipecat audio entry could abort STT earlier than the tap path intends.
- Phase-3 WebSocket/session scaffold (openSession, pipecat-tool.ts) is dormant but shipped; if pipecatServerUrl is ever set it points at a Railway/Python server whose availability and the 30s idle-drop behavior are untested in this codebase.
