# Camera / Vision capture-to-overlay pipeline (SmartMotion, Cage, on-device pose)

> Generated by QA audit 2026-07-21.

**Summary:** The camera/vision subsystem centers on app/swinglab/smartmotion.tsx (the unified swing-capture screen) and components/CageSessionOverlay.tsx (the canonical cage UI), both of which record with expo-camera's CameraView (with a staged react-native-vision-camera swap behind USE_VISION_CAMERA), run a PARALLEL expo-av acoustic recorder for strike/segment detection, then feed clips through a two-stage cloud+on-device analysis pipeline: (1) frame extraction via expo-video-thumbnails + expo-image-manipulator with AI swing-localization for untrimmed clips, sent to /api/swing-analysis for vision fault reads (services/poseDetection.ts); and (2) pose/biomechanics via analyzePoseFromUri which tries on-device MediaPipe (BlazePose 33→COCO-17) first and falls back to the /api/pose-analysis cloud proxy (services/poseAnalysisApi.ts + services/mediaPipePoseService.ts). Overlays (SwingBodyOverlay skeleton, ball/club arcs) are drawn frame-normalized then mapped into the video container's cover-space via services/swing/overlayCoords.ts. NOTE: app/smartvision.tsx is NOT a camera screen — it is the on-course satellite/aerial hole-view + GPS shot-tracking screen (its "vision" is api/hole-scan satellite geometry, not the device camera). Permissions are gated by a one-time preflight (services/permissionsManager.ts) plus per-screen camera/mic checks.

## 2. SmartMotion capture pipeline (the main flow)

File: `app/swinglab/smartmotion.tsx`. Phases: `setup → recording → analyzing → review` (`type Phase`, ~line 215).

### 2a. Two camera engines behind one ref
The screen keeps a single `cameraRef` and renders **one of two** camera engines (render block ~line 3507–3548):
- **Default:** expo-camera `<CameraView mode="video" mute facing={facing} mirror={false}>`. It is deliberately **`mute`** — the camera's own audio track is disabled because a SEPARATE `expo-av` Audio.Recording owns the mic for acoustic strike metering, and on iOS the audio session is a singleton (two recorders collide → silent metering loss).
- **Staged swap:** `components/capture/SwingVisionCamera.tsx` (react-native-vision-camera) when `useVisionCamera` is on. It mimics `CameraView`'s imperative `recordAsync()/stopRecording()` ref API so every `cameraRef` call works unchanged. Records **video-only** (`audio={false}`) at high fps (`PREFERRED_CAPTURE_FPS = 120`, via `useCameraFormat` prioritizing fps then max resolution) to give SmartTrace a dense ball-departure launch window.
- Flag: `services/capture/captureFlags.ts` — `DEFAULT_USE_VISION_CAMERA = false`; live value is a persisted runtime toggle in `store/captureEngineStore.ts` (owner flips it on `app/native-modules-debug.tsx`). **Only effective in a native build that linked vision-camera** — an OTA bundle falls back to expo-camera.
- `onCameraReady` on both engines drains `pendingStartRef` so a voice-requested "record" auto-starts once the camera (re)mounts coming out of review (hands-free loop).

### 2b. Recording (`startRecording`, ~line 2437)
1. Mic-permission gate: if `!micPerm?.granted`, request; on denial `Alert.alert('Microphone needed', …)` and bail.
2. Silences the caddie (`voiceService.stopSpeaking()`) so TTS audio never lands in the swing clip's audio track / flips the audio session mid-record.
3. **Prewarms the fault-read Lambda** (`swingAnalysisWarmup.prewarmSwingAnalysis`) — the open ≤60s record window is free warm time so the first swing lands on a hot Lambda (no cold-start "NO READ").
4. Clears prior swing state, `setPhase('recording')`, `setSmartMotionRecording(true)` (mic now belongs to the camera; taps mean STOP not voice-listen).
5. **Mode-aware capture** (read fresh from `useSettingsStore.environmentMode`, forced to `course` when `useRoundStore.isRoundActive`):
   - `cage` → parallel metered audio track → acoustic multi-strike segmentation.
   - `range` → metered track ALSO runs, but strikes are **candidates** the video locator confirms (a busy range's mic can't separate a neighbor's strike; the in-frame video can).
   - `course` → acoustics OFF (wind), single shot via video localization.
   - `chipSensitivity` inverts range/course metering (a quiet chip is unreadable over a noisy range).
   - `maxSec` = `RANGE_RECORDING_MAX_SECONDS` for range else `RECORDING_MAX_SECONDS`.
6. Parallel metering: `startMeteredRecording(cb)` from `services/swing/audioMetering.ts`; callback throttled to ~120 ms into `setLiveDb` (raw ~50 ms ticks would re-render the 4800-line component 20×/s). `meteringActive` reflects whether a mic track is truly running.
7. Assigns `recordingPromiseRef.current = cameraRef.current.recordAsync({ maxDuration: maxSec })` BEFORE arming the auto-stop timer (avoids the stop race). A `setTimeout(maxSec*1000)` sets `autoStopAtLimitRef` and calls `stopRecordingRef.current()` (the ref indirection guarantees the CURRENT `stopRecording`, not a stale closure).

### 2c. Stop → strikes → segments (`stopRecording`, ~line 2556)
1. Guards double-stop (`stoppingRef`), clears timers, `cameraRef.current?.stopRecording()`, `setPhase('analyzing')`.
2. Stops the metered recorder — the **only unbounded await** in the handoff, so it's `Promise.race`d against an 8 s fallback (expo-av `stopAndUnloadAsync` can hang; on timeout it loses only the acoustic read and degrades to whole-clip video analysis, never a permanent "Analyzing…" spinner). Captures `{ samples, uri, durationMs }`; `meteredDurationMs` is reused downstream so `probeDurationMs` never re-runs.
3. `detectStrikes(samples, opts)` (`services/swing/strikeDetector.ts`): env-class-matched calibration (`store/acousticCalibrationStore`), `CHIP_STRIKE_THRESHOLD_DB` when chip on, `noisyFloorDb` relaxed for range. Noisy-bay bail is guarded so a 3–5-swing recording never collapses to "1 of 1".
4. Strikes → `segmentsFromStrikes` / `segmentsFromVideoSwings` / `correlateStrikesWithVideo` / `filterReboundStrikes` (`services/swing/swingSegmentation.ts`) → `SwingSegment[]` (a reel to scrub). Cage finalizes from acoustics; range confirms candidates with the video locator.
5. Calls `runAnalysis(uri, segment)`.

### 2d. Ball-area auto-snapshot (expo-camera only)
During recording, `cameraRef.current?.takePictureAsync?.({ base64, skipProcessing })` (~lines 860, 2933, 2986) grabs still frames for ball-area anchoring / club scan. The vision handle lacks `takePictureAsync`, so those `?.` calls no-op there and ball-area falls back to manual anchoring.

## 3. Analysis pipeline — `runAnalysis` (~line 1489) + `services/poseDetection.ts`

`runAnalysis(rawUri, segment?)` orchestrates **two parallel tracks** with strong anti-poisoning guards (`sessionRunRef` run-token; every apply gated on `myRun === sessionRunRef.current`) and a 130 s `hangGuardMs` race so the screen can never stick on "Analyzing…".

### Track A — Vision fault read (`analyzeSwing`, poseDetection.ts line 744)
Started on the **raw** recorder file immediately (in parallel with the durable byte-copy `persistClipToDocuments`, a prior latency source). Context threaded in: club, swing #, caddie persona, angle (`down_the_line`/`face_on`/`glasses_pov`), handedness, language, CNS priors (`caddieMemoryStore` tendencies), player_context (handicap/miss/name), drill focus, `ball_area_norm`/`target_norm`, `tier:'quick'`, and a **`measured` block** (tempo ratio, backswing/downswing ms, shoulder tilt, spine delta, weight shift, launch divergence, peak dB) so the vision model corroborates against the app's own instrument readings.

Inside `analyzeSwing`:
1. **Duration probe** — `probeDurationMs` (Audio.Sound primary, `expo-video-thumbnails` upper-bound fallback, 8 s cap, `Promise.race`d).
2. **Swing localization (acoustics-free)** — for unbounded clips ≥ `LOCATE_MIN_CLIP_MS` (6 s): `locateSwingWindow` sends coarse timestamped small frames (`extractCoarseFrames`, 512px/q0.65, ~1/3 s clamped 10–16) to `/api/swing-analysis` with `mode:'locate_swing'`; the model returns a best-estimate `swing_time_sec` → a tight `[t-2.5, t+3]` window so dense frames land ON the swing. Range mode uses `locateSwings` (`mode:'locate_swings'`, denser, `mergeSwingDetections` collapses over-detections). Best-effort, never throws, never trips the breaker.
3. **Frame extraction** — `extractKeyFrames(clipUri, boundaries, quickTier, knownDurationMs)`:
   - Fractions: `FRAME_TIME_FRACTIONS = [0.08,0.40,0.60,0.75,0.88]` (impact-clustered); `QUICK_TIER = [0.10,0.55,0.85]` (3-frame, 512px/0.55 compress) for SmartMotion/Cage speed path; `LONG_CLIP` duration-scaled 6–12 frames spread across untrimmed uploads; medium clips back-window the last 5 s.
   - Each frame: `VT.getThumbnailAsync` (serialized wrapper `utils/videoThumbnail` to dodge the native retriever crash; retries at ±250 ms + 0 for VFR/Android MediaMetadataRetriever nulls) → `expo-image-manipulator` resize+JPEG+base64.
   - Empty extraction logs `frame_extraction_empty` to `issueLogStore` (uri scheme/tail/errors) for field diagnosis.
4. **POST `/api/swing-analysis`** `{ frames, context }` with `REQUEST_TIMEOUT_MS = 63_000` (must exceed the server's 60 s maxDuration + Anthropic→OpenAI→Gemini fallback chain; a shorter client abort was being mislabeled as a network loss). Fast-fail retry once on <10 s transient network blips; tier=quick 502/503 auto-retry once (warm-Lambda recovery) capped at 30 s. Returns `SwingAnalysisResult` (`ok`/`no_frames`/`no_network`/`error`) with structured `primary_fault`, cause/fix/drill, `evidence`, `strengths`, `contact_read`, `valid_swing`, and fault-frame URIs (wire + display quality). `analyzeSwingTentative` is the reduced-frame fallback (`TENTATIVE_TIMEOUT_MS = 55_000`).

### Track B — Pose / biomechanics (deferred to review, ~line 1873)
Runs in a `useEffect` gated on `phase === 'review'` (async, never blocks replay), **windowed to the selected swing** and **strike-anchored**:
- `extractPoseFramesFromVideo(clipUri, videoDurationMs, trustDuration=true, poseWindow, acousticImpactMs)` (`poseAnalysisApi.ts` line 641). When a real acoustic strike exists (`peakDb !== 0`), phases anchor to it: `P1_address = impact-2000`, `P2_takeaway = impact-1200`, `P4_top = impact-320`, `P6_impact = impact`, `P10_finish = impact+700` (fixes "top mid-backswing / impact 100 ms past ball"). Otherwise windowed fractions, else tiered whole-clip sampling.
- **On-device densification:** when `isMediaPipeAvailable()`, densifies from 5 anchors to ~20 frames across the span (~100–300 ms/frame on-device) for a smooth skeleton; cloud-only installs keep 5 (5–15 s/frame makes density unaffordable).
- `computeBiomechanicsFromFrames(frames, angle, handedness)` → `SwingBiomechanics` (shoulder/hip turn, tilt, spine delta, weight shift, each with `metric_confidence`). Committed to the session (`setSessionBiomechanics`) **only for the primary swing (index 0)** so scrubbing a multi-swing reel can't overwrite the saved report with the last-viewed swing.
- Tempo: `deriveSwingTempo` (also pose-based via `analyzePoseFromUri`).
- Motion sampler for the newer pose-first arc: `services/swing/poseMotionSampler.ts` `sampleSwingMotion` densely samples `analyzePoseFromUri` and returns wrist-centroid `MotionSample[]` (`poseMotion.ts` pure math, unit-tested separately).

## 4. On-device pose (MediaPipe) wiring

The **seam** is `services/poseAnalysisApi.ts::analyzePoseFromUri(imageUri, timestampMs)` (line 176):
1. **On-device first:** dynamic `import('./mediaPipePoseService')` → `detectPoseFromUri` → returns a COCO-17 `PoseFrame` or null.
2. **Cloud fallback:** if null/unavailable, POST base64/imageUrl to `/api/pose-analysis` (30 s timeout). Server returns `200 {configured:false}` when `POSE_API_KEY/HOST` unset → collapses to null (biomech card hides, no false 503).

`services/mediaPipePoseService.ts`:
- Wraps `NativeModules.MediaPipePose` (linked by `plugins/withMediaPipePose.js`; native impls `ios-native/MediaPipePoseModule.swift` + `android-native/MediaPipePoseModule.kt`; models `assets/mediapipe/pose_landmarker_{lite,full}.task`). Health-probed via `nativeModuleHealth`.
- `isMediaPipeAvailable()`, `detectPoseFromBase64/Uri`, `getMediaPipeStatus`, `setPreferredQuality`.
- **BlazePose 33 → COCO-17** projection (`BLAZEPOSE_TO_COCO17`, `projectBlazePoseToCoco17`); per-keypoint `score = min(visibility, presence)`.
- Quality `lite|full|heavy`, default `full`; `effectiveQuality` auto-drops to `lite` when `AppState !== 'active'` (background thermal throttle).
- `postProcessForGlassesPOV` zeros torso-joint scores when avg shoulder+hip visibility < 0.30 (Meta-glasses head-mounted POV has no torso in frame) so downstream biomech skips hip-turn/coil cleanly; arms/hands stay usable.
- `smoothPoseFrames` picks the best-scoring keypoint per joint across frames (composite skeleton).
- Other pose consumers: `services/poseEstimator.ts`, `services/swing/framingCheck.ts` (full-frame/ball-box framing coach), `services/swing/poseMotion.ts`.
- Build note: on-device pose needs a fresh native build (`GITHUB_TOKEN`); OTA on an expo-camera build won't have the native module — the seam degrades to cloud/null.

## 5. Overlay rendering

Used at smartmotion line 1139.
- `components/swinglab/SwingBodyOverlay.tsx` — skeleton overlay from `poseFrames`; interpolates between anchors; `faultJointsFor(faultId)` / `FAULT_REGION_JOINTS` highlight the joints implicated by the detected fault. Rendered ~line 3613.

- Body metric badges (`shoulder`, `hip`, etc.) render `~` prefix when `metric_confidence < 0.5` (honest low-confidence marking).

## 9. Data-flow summary (device-camera path)

```
[CameraView / SwingVisionCamera]  +  [expo-av Audio.Recording (metering)]
        │ recordAsync()                    │ startMeteredRecording
        ▼                                  ▼
   clip file:// (mp4)              MeterSample[] ─► detectStrikes ─► SwingSegment[]
        │                                                              │
        ▼ persistClipToDocuments (durable)                            │
   ┌──────────────────────── runAnalysis(uri, segment) ───────────────┘
   │  TRACK A (vision fault):                    TRACK B (pose, on review):
   │   probeDurationMs                            extractPoseFramesFromVideo
   │   locateSwingWindow (untrimmed)              (strike-anchored / windowed)
   │   extractKeyFrames (VT+manipulator)          analyzePoseFromUri
   │   POST /api/swing-analysis  ─► SwingAnalysis   ├─ on-device MediaPipe (BlazePose→COCO17)
   │                                                └─ else POST /api/pose-analysis
   │                                              computeBiomechanicsFromFrames ─► SwingBiomechanics
   ▼                                                        │
[Review UI] ── SwingBodyOverlay (skeleton) · ball/club arcs · badges
             frameToContainerNorm(..'cover') maps frame-norm → container
```


## Known risks
- USE_VISION_CAMERA path is default-OFF and only works in a native build that linked react-native-vision-camera; the vision handle lacks takePictureAsync so ball-area auto-snapshot silently no-ops (falls back to manual anchoring) whenever the vision engine is active.
- On-device MediaPipe requires a fresh native build (GITHUB_TOKEN); on an OTA/expo-camera build the native module is absent, so analyzePoseFromUri silently falls to the /api/pose-analysis cloud proxy — which returns {configured:false} when POSE_API_KEY/HOST are unset, collapsing to null and hiding all biomech/tempo. Pose can therefore be entirely dark in OTA builds with no visible error.
- iOS audio session is a singleton: the camera is recorded with mute and a separate expo-av recorder owns the mic. Any change that lets the camera capture audio, or that flips the audio session (e.g. TTS) mid-recordAsync, can silently kill acoustic metering → no strikes/segments/tempo. The voice path is under a standing FREEZE for this reason.
- The metered-recorder stop is the only unbounded await in stop→analyze and is Promise.race'd against 8s; if that guard were removed a hung stopAndUnloadAsync would strand the screen on 'Analyzing…' forever (the exact F1 cage bug it was added to fix).
- Cross-session poisoning is guarded by sessionRunRef run-tokens and a 130s hangGuard; the primary runAnalysis path historically lacked this guard, letting a slow read from session A land its verdict/report/CNS on session B. Any new async apply site in runAnalysis must re-check myRun === sessionRunRef.current.
- Client REQUEST_TIMEOUT_MS (63s) must stay above the server's 60s maxDuration + fallback chain; if lowered below it, slow-but-healthy analyses get aborted client-side and mislabeled as network loss (recurring historical bug).
- Frame extraction depends on expo-video-thumbnails / Android MediaMetadataRetriever, which returns null on variable-frame-rate phone clips at arbitrary timestamps; mitigated by a serialized VT wrapper + ±250ms/0 retries, but empty extraction still yields no_frames (clip 'plays but won't analyze').
- Biomech is committed to the saved session only for the primary swing (index 0); the live review UI updates per selected swing. Removing the index===0 guard reintroduces the bug where scrubbing a multi-swing reel overwrites the saved report with the last-viewed swing's skeleton/numbers.
- app/smartvision.tsx is easily mistaken for a camera screen given its name; it is the satellite/GPS aerial hole view. Work targeting the camera capture flow should go to smartmotion.tsx / CageSessionOverlay.tsx, not smartvision.tsx.
