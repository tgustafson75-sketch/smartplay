## TODO #1 — Sentry env wire-up
**Location:** `app/_layout.tsx:57`
**Marker:** `// TODO (Wednesday MacBook setup): add EXPO_PUBLIC_SENTRY_DSN + Sentry org/project to eas.json, then remove SENTRY_DISABLE_AUTO_UPLOAD=true from eas.json build profiles.`

**Context:** `Sentry.init` is gated on `process.env.EXPO_PUBLIC_SENTRY_DSN`. Without the DSN, Sentry is a no-op and `SENTRY_DISABLE_AUTO_UPLOAD=true` in `eas.json` skips the source-map upload step that otherwise needs Sentry org/project credentials to succeed.

**v1.2 unblock:**
1. Create / locate Sentry project; record DSN, org slug, project slug.
2. Add `EXPO_PUBLIC_SENTRY_DSN` to all three `eas.json` build profiles.
3. Add `SENTRY_ORG` and `SENTRY_PROJECT` to env (or `.sentryclirc`).
4. Remove `SENTRY_DISABLE_AUTO_UPLOAD=true` from `eas.json` profiles.
5. Re-build to confirm source maps upload cleanly.

**Severity:** MINOR. Without Sentry, crash reports surface only via adb logcat / device console. v1.1 ships with empirical Z Fold testing as the verification bar — Sentry wiring is a beta-distribution concern.

**Location:** `services/intents/queryStatusHandler.ts:563`
**Marker:** `const driverYards = 230; // TODO: read from accumulated club distances when wired`

**Context:** When the player asks "can I carry the bunker?" mid-round, the carry-feasibility calc assumes a fixed 230y driver. Once accumulated club-distance tracking lands, this should read the player's typical driver carry from observed shot data.

**v1.2 unblock:**
1. Add per-club distance accumulation (probably in roundStore or a dedicated clubStatsStore).

**Severity:** MINOR. The 230y default produces a usable answer for an average male recreational golfer; longer / shorter players get noticeably wrong answers. Not a v1.1 blocker because the underlying carry calc and voice response shape are correct.

## Web SSR / Zustand-rehydrate (F14) — SHIPPED 2026-05-05
**Location:** observed during Phase 100 F1 verification — Web bundle path
**Was:** `[roundStore] rehydrate error: ReferenceError: window is not defined` in Metro logs when launching the web target.

**Fix shipped:** `services/ssrSafeStorage.ts` — `getPersistStorage()` returns a noop storage shim during SSR (`typeof window === 'undefined'`) and the real AsyncStorage during native + client-side web. All 14 persisted Zustand stores updated to use it. SSR renders the in-memory default state; AsyncStorage hydration runs client-side after mount.
